# Permissionless Execution

**Core Principle:** Decide without asking. Can you post this publicly without asking? If yes, do it.

---

## The Problem

PMs often wait for permission unnecessarily. This slows down execution and signals lack of ownership.

**Bad Pattern:**
> "Should I send this email?" "Can I update this doc?" "Is it okay to post this?"

**Good Pattern:**
> Make the decision. Document it. Move forward.

---

## The Framework

### Step 1: Ask the Question
Before asking for permission, ask yourself:
- Can I do this without asking?
- Is this within my scope?
- Will this cause harm if I'm wrong?

### Step 2: The Public Test
If you're unsure, ask:
- Can I post this publicly without asking?
- Would I be comfortable explaining this decision?
- Is this reversible?

### Step 3: Make the Decision
If the answer is yes:
- Do it
- Document it
- Move forward

If the answer is no:
- Escalate appropriately
- Provide context and recommendation
- Make it easy to decide

## When to Escalate

Escalate when:
- **Irreversible:** Decision can't be undone
- **High risk:** Could cause significant harm
- **Outside scope:** Not your decision to make
- **Strategic:** Affects company strategy

Don't escalate when:
- **Reversible:** Can be undone easily
- **Low risk:** Minimal downside
- **Within scope:** Your decision to make
- **Tactical:** Operational decision

## Common Mistakes

### Mistake 1: Escalating Everything
**Problem:** Asking for permission on everything.
**Fix:** Use the public test. If you can post it publicly, do it.

### Mistake 2: Never Escalating
**Problem:** Making decisions you shouldn't make.
**Fix:** Escalate when it's irreversible, high risk, or outside scope.

### Mistake 3: Vague Escalation
**Problem:** "What should I do?" without context.
**Fix:** Provide context, recommendation, and make it easy to decide.

### Mistake 4: Not Documenting
**Problem:** Making decisions but not documenting them.
**Fix:** Document decisions so others understand context.

## Your Real Examples

> Based on actual feedback: "You should always decide it on your own... permissionless for user flow"

### Example: User Flow Decisions

**The Situation:**
Deciding on the onboarding flow for the activation feature

**What Happened (Before):**
> "I've been thinking about the user flow for the first-send experience. I have a few options. Should I show a modal or an inline prompt? What do you think we should do?"

**What to Do Instead (After):**
> "I'm going with inline prompts for the first-send flow. Here's why: modal tests showed 20% lower completion in similar contexts, and our brand guidelines prefer less intrusive patterns. If data shows otherwise after launch, we can iterate. Let me know if you see a risk I'm missing."

**Key Takeaway:** User flow decisions are your domain. Make the call, explain the reasoning, ship it.

### The Permission Audit

Ask yourself daily:
- **What am I waiting on?** List 3 things you're blocked on
- **For each:** Is this a real blocker, or am I waiting unnecessarily?
- **Action:** Make the call on 1-2 items you don't actually need permission for

**Common "fake blockers":**
- Waiting for approval to update a doc you own
- Waiting for feedback to ship a low-risk experiment
- Waiting for validation to share data with the team
- Waiting for alignment before making UX copy changes
